# Fine-Tuning

End-to-end fine-tuning pipeline: when to use it, dataset curation, job management, quality gates, and production integration.

---

## 1. When to Fine-Tune

Fine-tuning is the last resort for behavioral improvement. Apply it only when:

1. **The failure mode is reproducible** via an eval case (no eval coverage → don't tune yet)
2. **SPL and prompt engineering cannot close the gap** — try SPL updates first (they are cheaper and faster to iterate)
3. **The dataset is ≥50 high-quality examples** (OpenAI minimum; 200+ preferred for meaningful behavioral change)
4. **The behavior is stable** — tuning on a shifting target produces brittle models

**Do NOT fine-tune for:**
- One-off formatting preferences (use structured output / output format block)
- Behaviors that change per user or company (use SPL / cross_run_memory)
- Problems caused by bad context (fix context assembly first)
- Problems you can't eval (you won't know if it worked)

---

## 2. Dataset Quality Tiers

Eval scores determine which pipeline processes each example before inclusion in the training set:

| Eval score | Pipeline slug | What happens |
|------------|--------------|--------------|
| 0–20 | `ai_annotate_human_review_improve` | AI annotates → human reviews → AI improves → include |
| 20–40 | `ai_annotate_improve` | AI annotates → AI improves → include |
| 40–60 | `ai_improve` | AI improves response → include |
| 60–100 | `direct_include` | Include as-is (no processing needed) |

**Quality filter for dataset building:**
- `ground_truth = True` (operator-labeled golden example) → `direct_include`
- `eval_score >= 95` (from MetricObject) → `direct_include`
- Outcome high-quality IDs → `direct_include`
- Everything else → apply score-based pipeline

---

## 3. Outcome Scoring (for Dataset Selection)

Outcome scores are multi-component quality signals, not just LLM judge scores:

| Component | Weight | What it measures |
|-----------|--------|-----------------|
| `llm_judge` | 25% | LLM-evaluated quality of the response |
| `side_effect_completion` | 20% | Did the intended action succeed? |
| `tool_reliability` | 15% | Did tool calls succeed without retries/failures? |
| `hitl_resolution` | 15% | Did human-in-the-loop confirmations resolve correctly? |
| `feedback_followup` | 15% | Did the agent follow up correctly on user feedback? |
| `cost_latency` | 5% | Was the response within acceptable cost/latency bounds? |
| `evidence_retrieval` | 5% | Was the retrieved evidence relevant and used? |

**Promotion eligibility:** an AICall is promotion-eligible when `OutcomeScoringResult.promotion_eligible = True` and status is in `{"scored", "rescored"}`.

```python
)

```

### Manual dataset construction

For capabilities with insufficient historical data, build the dataset manually:

```jsonl
{"messages": [
    {"role": "system", "content": "{system_prompt}"},
    {"role": "user", "content": "{user_input}"},
    {"role": "assistant", "content": "{ideal_output}"}
]}
```

**Golden example checklist:**
- [ ] Response is the ideal output, not just an acceptable output
- [ ] Covers the typical case (happy path)
- [ ] Covers at least one edge case per capability
- [ ] No PII or sensitive data in any example
- [ ] System prompt in the example matches the current production system prompt
- [ ] Reviewed by a domain expert (not just generated by another model)

## 5. Job Launch

```python
from services.learning.fine_tuning.job import launch_job

job_aicall = launch_job(
    dataset=dataset,
    base_model="gpt-5-mini-2024-07-18",   # default from OPENAI_FINE_TUNE_DEFAULT_BASE_MODEL
    hyperparameters={
        "n_epochs": 3,       # start here; increase if underfitting
        "batch_size": 4,     # default; adjust for dataset size
    },
    user_id=user_id,         # for webhook tracking
)
```

**Job payload structure (SFT method-based):**
```json
{
  "model": "gpt-5-mini-2024-07-18",
  "training_file": "{file_id}",
  "method": {
    "type": "supervised",
    "supervised": {
      "hyperparameters": {"n_epochs": 3, "batch_size": 4}
    }
  }
}
```

**Webhook tracking:** `launch_job` creates an `AICall` record. When the OpenAI webhook fires, it updates that record with the job status.

---

## 6. Hyperparameter Selection

| Hyperparameter | Default | When to increase | When to decrease |
|---------------|---------|-----------------|-----------------|
| `n_epochs` | 3 | Underfitting (train loss still falling) | Overfitting (train loss << val loss) |
| `batch_size` | 4 | Large dataset (>1k examples) | Small dataset (<100 examples) |
| `learning_rate_multiplier` | auto | Model is slow to adopt new behavior | Model overshoots and forgets prior knowledge |

**Signs of overfitting:**
- Training loss falls but eval score on held-out set doesn't improve
- Model refuses to generalize beyond training examples
- Original capability on unrelated tasks regresses

## 7. Post-Training Eval Gate

After a fine-tuning job completes, require:

1. Run the same eval slugs that were used as baseline
2. New score ≥ baseline + 5 pts (meaningful improvement)
3. New score on unrelated capabilities ≥ baseline − 2 pts (no regression)
4. If any `hard_fail` appears post-tuning → rollback plan activates

```python
# Post-training gate
result = runner.run_slugs(
    slugs=affected_slugs,
    mode="standard",
)
delta = result["ci_gate"]["score"] - baseline_score
if delta < 5:
    raise ValueError(f"Fine-tuning did not improve by required margin: {delta:.1f} pts")
```

## 8. Scheduler & Automation

The fine-tuning scheduler runs automatically when:
- Dataset size threshold is crossed (`OPENAI_FINE_TUNE_DATASET_MIN_SIZE = 50` rows)
- A Celery periodic task triggers (`AutomationSubtype.FINETUNING`)
- An operator manually triggers via automation management

**Scheduler guard:** the scheduler will not launch a new job if a job for the same slug is already in progress or completed within the cooldown period.

## 9. Full Pipeline Summary

```
High-quality AICalls / Golden examples
        │
        ▼
Dataset quality filter (ground_truth OR eval_score >= 95 OR outcome high-quality)
        │
        ▼
Score-based pipeline tier (ai_annotate_human_review_improve → direct_include)
        │
        ▼
JSONL upload → OpenAI Files API (file_id)
        │
        ▼
Fine-tuning job launch (SFT, n_epochs=3, batch_size=4)
        │
        ▼
Webhook → AICall record updated
        │
        ▼
Post-training eval gate (delta ≥ 5 pts, no regression)
        │
        ▼
Deploy fine-tuned model slug
        │
        ▼
Monitor production (eval scores, latency, cost)
```

## 10. Integration with SPL

After a fine-tuning job produces a better-behaved model, update SPL blocks to reflect the new behavior baseline. Do not use fine-tuning as a workaround for missing SPL blocks — they serve different purposes:

| Mechanism | Scope | Speed | Cost |
|-----------|-------|-------|------|
| SPL (system prompt block) | Explicit instruction | Instant | Zero |
| Cross-run memory | User/company preference | Instant | Zero |
| Fine-tuning | Baked-in capability | Hours + training cost | High |

**Use SPL first. Use fine-tuning only when SPL cannot achieve the required behavior reliably.**
